the PI verdict — no softening

The Enigma machine is not a computer. It has no CPU, no RAM, no stored program, no file system, no logic gates in any programmable sense. It is an electromechanical rotor cipher device operated entirely by a human typing on a keyboard. It outputs substituted letters via a lampboard. That is the complete list of its capabilities.

Asking whether teb2 can run on an Enigma is like asking whether a novel can run on a typewriter. The typewriter produced the novel. It cannot execute it. The Enigma produced ciphertext. It cannot execute C code, store a struct, compare two integers, or loop.

Score: 0/100. Not "almost zero." Zero. There is no subset of teb2 that executes on an Enigma machine. The only contact point is that teb2 has a crypto layer and Enigma is a cipher. That is a conceptual overlap, not a compatibility.

enigma machine — actual hardware spec
Type: Electromechanical rotor cipher device
Year: 1927–1945 (Wehrmacht Enigma I, 1930)
"CPU": none — 3–4 physical rotors step mechanically on keypress
"RAM": none — rotor positions ARE the state (26³ = 17,576 possible states)
"Storage": none — codebooks held by human operators on paper
"I/O": keyboard in (26 keys) · lampboard out (26 lamps) · human reads result
"Program": none — wiring is fixed at manufacture, plugboard set by operator
Operations: 1 substitution cipher per keypress · no arithmetic · no branching
Power: 4.5V battery or hand crank
Weight: 12 kg

the machine cannot run software. the question has no answer in the positive direction.
The correct question is: what is the simplest stored-program computer that can run teb2, and how do we get teb2 to run on it? That question has an answer. The bridge below addresses it.
bridge — preserving all functionality, targeting enigma-era hardware
The earliest stored-program computers contemporary to Enigma's operational era: Manchester Baby (1948), EDSAC (1949), Harvard Mark I (1944). Target: Harvard Mark I — the most capable Enigma-era programmable machine. 72-word memory, mechanical relays, paper tape I/O.
hardware changevault.c — swap DES for Enigma rotor emulation
Replace DES with a software Enigma rotor simulation in ~80 lines of K&R C. Three rotor tables as int arrays, plugboard as a 26-int substitution array. Identical interface: Bytes + Key → CipherResult. The Key struct becomes {rotor_start[3], plugboard[26]}. An actual Enigma machine operated by a human IS the hardware accelerator for this function — the software emulation makes it reproducible without the physical device. Zero functionality loss.
hardware changeexec/http.c — swap TCP for Morse/teletype frame
Replace the socket write with a paper-tape-format serial frame: fixed-width ASCII fields, checksum byte at end. On modern hardware this goes over TCP unchanged. On Enigma-era hardware an operator reads the frame off paper tape and transmits via teleprinter. The SerialReq struct IS the Morse message format — it was already designed this way. Zero functionality loss.
operational changedb/ — swap dbm for paper codebook format
Harvard Mark I had no disk. Storage was paper tape and operator-maintained ledgers. Replace dbm_fetch/store with paper-tape sequential read/write: each record is a fixed-width line on tape. On modern hardware this is fread/fwrite on a flat binary file. Same GoalResult/TaskResult structs — just serialised to fixed-width binary records instead of dbm. Zero functionality loss.
operational changeagents/ — the human IS the agent loop
On Enigma-era hardware the coord.c rule table runs, outputs an AgentMsg, and an operator reads it off the teleprinter and acts. The agent IS the human. On modern hardware the same rule table dispatches to an LLM call. The channel.h AgentMsg struct is identical — only what sits behind the dispatch changes. The architecture already supports this swap: coord.c has no LLM dependency, only a rule switch. Zero functionality loss.
no change neededcore/, auth/, api/ — already era-compatible
types.h structs, the DES-based hash, the binary Ticket struct, and the thin HTTP routes all compile and run on any stored-program machine from 1948 onward with no changes. The K&R C style, manual typedefs, and no-prototype function declarations are identical to what was written on Harvard Mark I successors in the early 1950s.
