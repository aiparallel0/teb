8. Cross-Cutting Summary of Architectural Anti-Patterns
The "Accumulative PR" Pattern
Every PR appended to the 3 god-files instead of extracting. The typical PR structure:

Append new dataclasses to models.py (bottom)
Append new CRUD functions to storage/_monolith.py (bottom)
Append new endpoint handlers to main.py (bottom)
Append a new test file covering only the new feature
No PR ever deleted or refactored existing code (except PR#40 which removed 8 duplicate tests, and PR#41 which split storage — but that split is now circular). The codebase grew in one direction: down.

The "Grade-Driven Development" Trap
PR#18 is literally titled "Fix missing parts to increase average grade to A-", and PR#29 was created based on "grading the repo against the agent spec." This suggests the AI agent was being scored on feature coverage, incentivizing adding more code rather than improving quality.

The "Empty PR" Problem (PR#36)
One PR produced zero code changes after the agent timed out planning. This reveals that agents are solving complex tasks that require more context (touching 5,000+ line files) than fits in a session.

The "Authorization Afterthought" Pattern
18 review comments on PR#42 identified missing authorization checks on content blocks and custom fields. These are security vulnerabilities where any authenticated user can read or modify any other user's data. The pattern repeats because endpoints are written by copy-paste from other endpoints, and the original templates don't always include ownership verification.

The "Brittle JS" Pattern
Four separate PRs (#26, #30, #32, #33) fixed Temporal Dead Zone (TDZ) bugs in app.js where const declarations appear after usage due to JavaScript hoisting semantics. This is a structural problem: a 5,953-line monolithic JS file where declaration order matters and is frequently violated during feature addition.

The "Fix What Was Just Added" Pattern
PR#20 ("AI never called — $0 spend") revealed the LLM was never being invoked despite the entire product being built around AI. PR#39 ("Fix broken API endpoints") found 8 frontend calls going to non-existent endpoints. PR#44 fixed a broken deploy pipeline. These rapid bug-fix PRs after feature PRs suggest features are being added without integration testing.

9. What the New Repository Must Avoid
Based on this audit, the core bottleneck is the god-file pattern amplified by append-only development. Specific prescriptions:

No file should exceed 500 lines. main.py at 8,272 lines consumes the full context window of any model. Domain router files (goals, tasks, payments, etc.) must be separate.

storage/_monolith.py must not exist. Generic CRUD via a BaseRepository[T] class + per-entity subclasses eliminates 65 _row_to_* functions and enables AI agents to reason about one domain at a time.

conftest.py must contain all fixtures. The 14+ duplicated _fresh_db/setup_test_db/_register_and_login/_make_goal/_make_task helpers must live in one place.

Circular imports must be structurally impossible. The layered architecture must be: config → models → storage → services → routers → main. Nothing upstream may import downstream.

Authorization must be enforced at the router layer before any business logic. Not as an afterthought inside handler functions.

app.js must be an ES module with imports. The 5,953-line vanilla JS file cannot be maintained by AI agents. The partial extraction into views/ (7 files = 1,383 lines) shows the pattern works; it just wasn't completed.

CI must parallelize tests and reuse the Docker image from the test job in the deploy job. The current sequential pipeline adds unnecessary wall-clock time.

PRs must be scoped to a single domain. The pattern of touching models + storage + main + tests + frontend in one PR makes every PR a context-window-consuming mega-change.
