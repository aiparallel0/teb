hard walls — non-negotiable realities
Wall 1 — No C compiler. K&R C was invented in 1972 on a PDP-11. No C compiler exists for the IBM 701, has ever existed, or can practically be built. Every .c and .h file in teb2 must become IBM 701 assembly source. This is not a workaround. It is a complete language replacement.
Wall 2 — 9 KB RAM. The entire running program — code + data + stack — must fit in 9 KB at any moment. teb2's types.h alone describes structs totalling ~400 bytes. A single Goal struct with all fields is ~80 bytes. You can hold ~115 Goals in RAM simultaneously. Every module must be an overlay loaded from drum on demand and evicted when done.
Wall 3 — No OS, no filesystem, no sockets. There is no operating system. No file open/close. No TCP. I/O is punch cards (input) and magnetic tape reels (output/storage). The db layer becomes sequential tape record read/write. The HTTP layer becomes punched card batches.
Wall 4 — Self-modifying code for arrays. There are no index registers. Looping over an array requires modifying the address field of a load instruction at runtime. Every loop in every file must use this technique.
Wall 5 — No floating point. Hardware has none. All arithmetic is integer. Already handled — teb2 uses i64 cents. Keep it.
memory map — 9 KB budget allocated per module
IBM 701 RAM — 2,048 words × 36 bits (9,216 bytes total)

addr 0000–0063 ( 100 words / 450 bytes) resident: types + error constants
addr 0064–0127 ( 64 words / 288 bytes) resident: current module code (overlay)
addr 0128–0255 (128 words / 576 bytes) resident: current module working data
addr 0256–0511 (256 words / 1,152 bytes) resident: I/O buffer (tape record in/out)
addr 0512–0767 (256 words / 1,152 bytes) drum transfer buffer
addr 0768–2047 (1,280 words/ 5,760 bytes) available: next overlay pre-load area

RULE: one module loads at a time. finish, write result to drum, evict, load next.

Becomes auth/hash.asm — Enigma rotor emulation in 701 assembly. Three rotor tables as 26-word arrays on drum. Self-modifying loop steps through each character. Input: password word-address + rotor-start word-address. Output: hashed word-address in accumulator.

auth/token.c

exec/vault.c

Both become exec/card.asm — output is a punched card batch. Each API call becomes a fixed-format card record written to the card punch. Input arrives from card reader at next job step. No TCP, no subprocess. One file replaces two because the interface is identical: SerialReq → punch → read → SerialResult.

Becomes agents/channel.asm — AgentMsg is 4 words: tag word (36-bit enum), payload words 1–3. MSG_GOAL_NEW EQU 1, MSG_TASK_DONE EQU 2, etc. Message queue is a circular drum segment: 64 records × 4 words = 256 drum words.

Each becomes a .asm overlay. Loaded from drum when a message for that agent arrives. Dispatch is a jump table: load msg tag, multiply by 2 (word offset), self-modify a branch instruction address, execute. Output: new AgentMsg written to drum queue. Evicted after processing.

api/goals.c

api/auth.c
Become card-reader handlers. Each "HTTP request" is a punched card with fixed-format fields. The api module reads the card, validates the ticket word, dispatches to db or auth overlay. main.c
Becomes main.asm — the boot loader. Reads config tape record, initialises drum queue, enters card-read loop. Dispatch loop: read card, extract module tag from first word, load that module overlay from drum to addr 0064, jump to it, on return write output card, loop. 40 instructions total.
rewrite
Makefile
Becomes assembly job control: run IBM 701 assembler (or modern cross-assembler targeting 701 binary) on each .asm file, produce binary card deck per module, load sequence documented. On modern hardware: use the open-source i701 emulator — it accepts 701 assembly and produces authentic binary output.

new file added — overlay loader
new filecore/loader.asm — drum overlay manager
The 9 KB wall forces this to exist. Resident at fixed address. Two functions: drum_load(module_id, dest_addr) and drum_save(src_addr, module_id). Every module calls loader before jumping to another module. 20 instructions. This is the only file with no equivalent in the original teb2 — it exists solely because of the memory constraint.
